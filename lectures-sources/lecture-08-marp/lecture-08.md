---

# Коли гра починає «бачити»

**Класичний NPC** — отримує прямий доступ до координат, raycast, бінарне «бачу / не бачу». По суті — **читерить**.

**CV-NPC** — отримує зображення з камери, поступово розпізнає об'єкти, може помилятися. Поводиться **як людина**.

Три напрямки CV в іграх:

| Напрямок | Приклади |
|----------|----------|
| **Всередині гри** | NPC-зір, аналіз кадрів у реальному часі |
| **У розробці** | Автоматичне QA, пошук багів, класифікація контенту |
| **Навколо гри** | Аналіз стрімів, античіт, розпізнавання жестів |

---

# Задачі CV: від простого до складного

**Класифікація** — що на зображенні?
**Детекція** — де саме кожний об'єкт? (bounding box)
**Сегментація** — яким пікселям належить кожний об'єкт?

> Зростає рівень деталізації: клас → рамка → маска пікселів

---

# Детекція об'єктів: формулювання

Для кожного об'єкта модель передбачає:

- **Координати рамки:** $(x, y, w, h)$
- **Клас об'єкта:** $c \in \{1, \ldots, C\}$
- **Впевненість:** $p(c) \in [0, 1]$

> **Складність:** десятки/сотні об'єктів різного розміру, з перекриттями, на складному фоні — за один прохід мережі

---

# Еволюція підходів до детекції

### Двоетапні (Two-stage)
- **R-CNN** (2014): Selective Search + CNN
- **Fast R-CNN**: CNN один раз на зображення
- **Faster R-CNN**: Region Proposal Network
- Точно, але повільно (~5 FPS)

### Одноетапні (One-stage)
- **YOLO**: сітка S×S, один прохід
- **SSD**: multi-scale feature maps
- **RetinaNet**: Focal Loss для балансу класів
- Швидко і точно (400+ FPS)

# Метрики: IoU та mAP

### IoU (Intersection over Union)

$$\text{IoU} = \frac{|B_{\text{pred}} \cap B_{\text{gt}}|}{|B_{\text{pred}} \cup B_{\text{gt}}|}$$

IoU ≥ 0.5 → об'єкт вважається «знайденим»

### mAP (mean Average Precision)
- Для кожного класу — крива precision-recall
- AP = площа під кривою
- mAP = середнє по всіх класах
- **mAP@50:95** — основний бенчмарк (середнє IoU від 0.5 до 0.95)

# U-Net: encoder-decoder з skip connections

**Архітектура:**
- **Encoder** зменшує роздільність → витягує високорівневі ознаки
- **Decoder** відновлює роздільність → локалізує межі
- **Skip connections** передають деталі з encoder до decoder

> Оригінально для медичних зображень (2015), але став стандартом для будь-якої сегментації

# Vision Transformer (ViT)

**Ідея:** замість згорток — механізм self-attention на патчах зображення.

**Пайплайн:** Зображення → патчі 16×16 → embedding + позиції → Transformer encoder

### Swin Transformer (2021)
- Локальні вікна → лінійна складність замість квадратичної
- Shifted windows між шарами для обміну інформацією
- Новий SOTA backbone для всіх задач CV

### DETR (2020)
- End-to-end детекція без anchor boxes і NMS
- Object queries → predictions через Hungarian matching

---

# CLIP: зір + мова

**OpenAI, 2021:** 400M пар «зображення + текст» → спільний embedding-простір

> Zero-shot класифікація: опишіть категорію текстом — CLIP знайде відповідні зображення без дотренування

### Застосування в геймдеві:
- **Семантичний пошук ассетів** — «mossy stone wall in forest» → знаходить з 50K текстур
- **Автокласифікація** — жанр скріншоту, настрій сцени
- **Генерація** — CLIP + Stable Diffusion = концепт-арт з тексту
- **Стилістична консистентність** — перевірка відповідності арт-стилю гри

---

# Від 2D до 3D: навіщо?

CNN розуміють зображення — але ігри та роботи живуть у **3D-просторі**.

Як отримати 3D-геометрію з плоских фото?

| Підхід | Суть | Компроміс |
|--------|------|-----------|
| **Датчики глибини** | LiDAR, стерео, ToF | Точно, але дорого |
| **Моноколярна оцінка** | Нейромережа за 1 фото | Швидко, але неточно |
| **SfM** | Геометрія з набору фото | Достатньо смартфона |

---

# Паралакс: як ми бачимо глибину

Людський зір: два ока, паралакс → відчуття глибини.

Камера що рухається = **віртуальне друге око**.

> **Ключова ідея:** якщо знати, де ті самі точки з'являються на різних фото, можна обчислити і положення камер, і 3D-координати точок.

Два погляди на одну точку → **тріангуляція** → 3D-координати

---

# Structure from Motion — що це?

> Одночасне відновлення **структури сцени** (3D-координати точок) та **руху камери** (положення + орієнтація) з набору неструктурованих фотографій

**Вхід:** набір звичайних фото (будь-який порядок)

**Вихід:** розріджена хмара 3D-точок + калібровані пози камер

---

# Інкрементальний SfM: кроки 1–3

### 1. Виділення ознак
SIFT / SuperPoint знаходять точки, інваріантні до масштабу, повороту, освітлення. Кожна точка → числовий дескриптор.

### 2. Зіставлення (Matching)
Порівняння дескрипторів між парами фото. Результат: пари відповідних точок.

### 3. Геометрична верифікація
RANSAC оцінює **фундаментальну матрицю** — геометричне відношення між двома видами. Помилкові пари (outliers) відкидаються.

> Результат: очищений **граф сцени** — які зображення пов'язані геометрично

---

# Інкрементальний SfM: кроки 4–7

### 4. Ініціалізація
Обрати пару фото з великою базою → тріангулювати початкові 3D-точки

### 5. Реєстрація нових камер
Для кожного нового фото: знайти відповідності з відомими 3D-точками → розв'язати задачу **PnP** (Perspective-n-Point) → отримати позу камери

### 6. Тріангуляція
Нова камера створює нові пари → обчислити нові 3D-координати → хмара зростає

### 7. Bundle Adjustment
Нелінійна оптимізація всіх поз і точок одночасно

> Кроки 5–7 **повторюються** для кожного нового зображення

---

# Bundle Adjustment

Мінімізація **помилки репроекції** — різниці між спостереженими 2D-координатами точки на фото і її проекцією з 3D-моделі:

$$\min_{\mathbf{X}, \mathbf{P}} \sum_{i} \sum_{j} \| \mathbf{x}_{ij} - \pi(\mathbf{P}_j, \mathbf{X}_i) \|^2$$

- $\mathbf{X}_i$ — 3D-координати $i$-тої точки
- $\mathbf{P}_j$ — параметри $j$-тої камери
- $\pi$ — функція проекції 3D → 2D
- $\mathbf{x}_{ij}$ — спостережувані 2D-координати

> Найдорожчий крок обчислювально, але забезпечує точність всієї реконструкції

# SfM у навігації дронів

### До польоту (SfM — офлайн)
- Зйомка з перекриттям → SfM+MVS на сервері
- Результат: ортофотоплан + Digital Surface Model
- Інструменти: Pix4D, OpenDroneMap, DJI Terra

### Під час польоту (SLAM — реальний час)
- Visual-Inertial Odometry (VIO) + SLAM: камера + IMU, 30+ Hz
- ORB-SLAM3, VINS-Mono
- SfM-карта використовується як *апріорна* інформація

> **Проблема масштабу:** монокулярний SfM не знає реальних розмірів. 1 м чи 10 м? Для дрона це критично. Розв'язання: GPS-мітки, IMU, відомі об'єкти.

# Нейронні методи: NeRF та 3D Gaussian Splatting

### NeRF (Neural Radiance Fields)
Нейромережа вивчає функцію: $f(\mathbf{x}, \mathbf{d}) \to (c, \sigma)$
Координата + напрямок → колір + щільність
Тренування: години. Рендер: секунди/кадр.

### 3D Gaussian Splatting (3DGS)
Набір 3D-еліпсоїдів з кольором, диференційована растеризація.
Тренування: 30–60 хв. **Рендер: 90+ FPS.**

> Обидва методи використовують **COLMAP-пози** як вхід. SfM дає структуру — нейронні методи дають зовнішній вигляд.

# 3DGS для ігор: можливості та обмеження

### Чого ще немає:
- Полігональний меш → немає колізій
- Фізика — об'єкти не взаємодіють
- LOD — складно оптимізувати
- Обмежене редагування

### Де вже працює:
- Кінематографічні фони, VR-середовища
- Швидкий прототипінг рівнів, превізуалізація

> **glTF:** розширення `KHR_gaussian_splatting` (серпень 2025)
> **UE5:** плагін XVERSE XScene (Apache 2.0)

# Гібридний конвеєр (best practice)

```
Відео з телефону/дрона → COLMAP / RealityCapture (пози)
  ├── PostShot / Nerfstudio → 3DGS (візуальний фон)
  └── COLMAP dense / OpenMVS → меш (колізійна геометрія)
Обидва імпортуються в UE5
```

> 3DGS для красивої картинки + класичний меш для фізики та взаємодії

# Реконструйовані середовища для RL

**Проблема:** RL-агенти потребують тисяч годин у 3D-середовищі. Моделювати рівні вручну — дорого.

**Рішення:** відсканувати реальну будівлю → готове середовище за годину.

| Платформа | Опис | Дані |
|-----------|------|------|
| **Meta Habitat** | Швидкий 3D-симулятор | HM3D: 1000 сканів будівель |
| **Replica** (Meta) | Високоякісні скани з семантикою | 18 приміщень, HDR + розмітка |
| **iGibson** (Stanford) | Інтерактивні сцени з фізикою | Готовий RL-код (SAC) |
| **AirSim** | UE5 як бекенд для RL | Будь-який UE5-рівень |

---

# Від скану до тренованого агента

1. **RGB-D скан** (або SfM+MVS) → меш
2. **Семантична анотація** — кімнати, об'єкти, прохідні зони
3. **Завантаження в симулятор** → RL-агент тренується:
   - Point-goal navigation
   - Object-goal navigation
   - Instruction following

### Для геймдеву:
- Скан будівлі → бот тренується **до** художнього polish
- RL-агент проходить рівень 1000 разів → метрики для левел-дизайнерів

> **SLAM3R** (CVPR 2025): реконструкція з відео через feed-forward мережу — без SfM

# Повний пайплайн

```
Зображення
  → YOLO         → детекція об'єктів (що і де)
  → DeepLab/SAM  → сегментація (піксельні маски)
  → Depth Anything → оцінка глибини (3D з одного кадру)
  → SfM + MVS    → 3D-реконструкція (з набору фото)
  → Fusion       → 3D-позиції, прохідні зони, контекст
  → Ігрова логіка
```

> Детекція + сегментація + глибина + реконструкція = **повне розуміння сцени**

---

# Ключові висновки

1. **YOLO** — реалтайм детекція для NPC-зору, QA, античіту
2. **Сегментація** (U-Net → SAM) — попіксельне розуміння для процедурної генерації та AR
3. **Vision Transformers** — глобальний контекст замість локальних згорток
4. **CLIP** — зв'язок зору та мови для zero-shot і генерації
5. **SfM** — фундамент 3D-реконструкції: фотографії → хмара точок + пози камер
6. **NeRF / 3DGS** — фотореалістичний вигляд, але без мешу для фізики
7. **Реконструйовані скани** → ігрові рівні + тренування RL-агентів
